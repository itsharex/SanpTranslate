## 2. 技术栈选型

### 2.1 选型原则

| 原则       | 说明                                                         |
|-----------|--------------------------------------------------------------|
| 轻量化     | 安装包体积小，运行时资源占用低                               |
| 跨平台     | 一套代码支持 Windows / macOS / Linux                         |
| 原生性能   | 截图、OCR 等系统级操作需接近原生性能                         |
| 安全性     | 密钥安全存储，通信加密，无遥测                               |
| 可维护性   | 模块低耦合，核心能力可独立更新                               |

### 2.2 技术栈总览

| 层级           | 技术                          | 版本       | 用途                           |
|---------------|-------------------------------|------------|--------------------------------|
| 桌面框架       | Tauri                         | 2.x        | 跨平台桌面应用壳               |
| 前端框架       | Vue 3                         | 3.5+       | UI 渲染与交互                  |
| 前端语言       | TypeScript                    | 5.x        | 类型安全的前端开发             |
| UI 组件库      | Naive UI                      | 2.x        | 轻量级 Vue 3 组件库            |
| 构建工具       | Vite                          | 6.x        | 前端构建与热更新               |
| 后端语言       | Rust                          | 1.85+      | 系统级操作与核心逻辑           |
| 屏幕截图       | xcap                          | 0.9+       | 跨平台屏幕捕获                 |
| OCR 引擎       | Tesseract (via leptess)       | 5.x        | 离线文字识别                   |
| 数据库         | SQLite (via rusqlite)         | 3.x        | 历史记录本地存储               |
| 安全存储       | keyring                        | 2.x        | OS 凭据管理器加密存储密钥      |
| HTTP 客户端    | reqwest                       | 0.12+      | 大模型 API 通信                |
| 序列化         | serde / serde_json            | 1.x        | 数据序列化与反序列化           |
| 剪贴板         | tauri-plugin-clipboard-manager| 2.x        | 系统剪贴板读写                 |
| 全局快捷键     | tauri-plugin-global-shortcut  | 2.x        | 全局快捷键注册与监听           |
| 状态管理       | Pinia                         | 2.x        | Vue 前端状态管理               |
| 国际化         | vue-i18n                      | 10.x       | 界面语言切换                   |

### 2.3 选型理由

#### 2.3.1 Tauri vs Electron

| 对比维度     | Tauri 2.x                          | Electron                        |
|-------------|-------------------------------------|---------------------------------|
| 安装包体积   | ~5-15 MB（不含 OCR 数据）           | ~150 MB+                        |
| 运行时内存   | ~30-80 MB                           | ~150-300 MB                     |
| 系统集成     | Rust 原生调用，性能接近原生          | Node.js 桥接，有性能损耗        |
| 安全模型     | 默认最小权限，IPC 白名单             | 完整 Node.js 运行时，攻击面大   |
| 跨平台       | Windows / macOS / Linux             | Windows / macOS / Linux         |
| 生态成熟度   | 快速成长中，核心插件齐全             | 非常成熟，插件丰富              |

**结论**：SnapTranslate 定位为轻量级工具，Tauri 在安装包体积、运行时资源占用和安全性方面具有显著优势，且其 Rust 后端天然适合截图、OCR 等系统级操作，故选择 Tauri。

#### 2.3.2 Vue 3 vs React

| 对比维度     | Vue 3                               | React                           |
|-------------|-------------------------------------|---------------------------------|
| 学习曲线     | 模板语法直观，上手快                 | JSX + Hooks，概念较多           |
| 生态集成     | Naive UI 等轻量组件库成熟            | 组件库选择多但体积偏大          |
| 响应式系统   | Proxy-based，细粒度更新              | 不可变数据，需手动优化          |
| Tauri 集成   | 社区示例丰富                         | 社区示例丰富                    |

**结论**：Vue 3 的模板语法更直观，Naive UI 轻量且 tree-shakable，适合本项目的极简 UI 需求。

#### 2.3.3 Tesseract OCR 选型

| 方案               | 优势                           | 劣势                             |
|--------------------|--------------------------------|----------------------------------|
| Tesseract (leptess) | 成熟稳定，多语言支持，离线运行  | 需系统安装 Tesseract 库          |
| Tesseract (tesseract-rs) | 可静态编译，无运行时依赖  | 编译复杂，构建时间长             |
| Windows OCR API    | 原生集成，无需额外安装          | 仅 Windows 可用                  |
| PaddleOCR          | 中文识别精度高                  | Python 生态，Rust 集成困难       |

**结论**：选择 Tesseract OCR，通过 leptess crate 提供 Rust 绑定。Tesseract 是最成熟的开源 OCR 引擎，跨平台支持好，语言数据可独立更新。发布时采用动态链接方式，安装包内包含 Tesseract 运行时库。

## 3. 系统架构

### 3.1 架构总览

SnapTranslate 采用 **Tauri 双层架构**：前端（WebView）负责 UI 渲染与用户交互，后端（Rust）负责系统级操作与核心业务逻辑。两层通过 Tauri IPC 机制通信。

```
|  |  +------------------+  +------------------+                |  |
|  |                                                            |  |
|  |  +------------------+  +------------------+                |  |

|  |  | Overlay.vue       |  | HistoryView.vue  |                |  |
|  |  +------------------+  +------------------+                |  |
```

### 3.2 分层架构说明

#### 3.2.1 前端层（Presentation Layer）

**职责**：UI 渲染、用户交互处理、前端状态管理。

**关键设计决策**：
- 每个贴图实例对应一个独立的 Tauri 窗口，窗口内运行独立的 Vue 组件实例。
- 截图蒙版使用全屏透明窗口实现，由前端绘制选区矩形。
- 所有前端状态通过 Pinia store 管理，与后端通过 Tauri `invoke` 命令通信。

#### 3.2.2 后端层（Core Layer）

**职责**：系统级操作（截图、OCR、剪贴板、快捷键）、业务逻辑（翻译调度、历史管理）、数据持久化。

**关键设计决策**：
- 后端模块以 Rust crate 形式组织，每个模块暴露 Tauri Command 供前端调用。
- OCR 操作在独立线程执行，避免阻塞主线程。
- 翻译 API 调用使用异步 I/O（tokio），支持超时控制。
- 配置文件使用 TOML 格式，API 密钥通过 keyring crate 存入 OS 凭据管理器。

#### 3.2.3 通信层（IPC Layer）

**通信方式**：
- **前端 -> 后端**：`tauri::command`（invoke），用于请求数据或触发操作。
- **后端 -> 前端**：`tauri::event`（emit），用于推送状态变更（如翻译完成、OCR 进度）。

### 4.2 模块职责

| 模块       | 职责                                               | 关键依赖                     |

## 5. 数据流设计

### 5.1 截图贴图数据流

```
用户按下 Ctrl+Alt+L
        |
        v
[hotkey 模块] 捕获快捷键事件
        |
        v
[capture 模块] 捕获全屏图像
        |
        v
[window 模块] 创建全屏透明蒙版窗口
        |
        v
[前端 Overlay.vue] 用户拖拽框选区域
        |
        +---> 用户按 Esc 或右键点击 ---> 销毁蒙版窗口（取消截图）
        |
        v
[前端 -> 后端] invoke("capture_region", {x, y, w, h})
        |
        v
[capture 模块] 裁剪选区图像，返回图像数据(Base64)
        |
        v
[clipboard 模块] 将图像写入系统剪贴板
        |
        v
[window 模块] 创建贴图窗口(原位，无边框置顶)
        |
        v
[前端 PinWindow.vue] 显示截图图像与控制栏
```

### 5.2 OCR 翻译数据流

```
用户点击"AI翻译"按钮
        |
        v
[前端 -> 后端] invoke("start_ocr_translate", {pin_id, image_data})
        |
        v
[translate 模块] 判断翻译模式
        |
        v (OCR 模式)
[ocr 模块] 本地 Tesseract 识别
        |
        v
返回: [{text, x, y, w, h}, ...]
        |
        v
[translate 模块] 构造翻译 Prompt，调用文本大模型 API
        |
        v
[reqwest] POST -> 大模型 API -> 返回译文
        |
        v
[translate 模块] 解析译文，映射回各文本块
        |
        v
[后端 -> 前端] emit("translate_result", {pin_id, results})
        |
        v
[前端 PinWindow.vue] 渲染译文覆盖标签
```

### 5.3 多模态翻译数据流

```
用户点击"AI翻译"按钮
        |
        v
[前端 -> 后端] invoke("start_multimodal_translate", {pin_id, image_data})
        |
        v
[translate 模块] 图像编码为 Base64
        |
        v
[reqwest] POST (含图像) -> 视觉大模型 API -> 返回整体译文
        |
        v
[后端 -> 前端] emit("translate_result", {pin_id, full_text})
        |
        v
[前端 PinWindow.vue] 显示"译文面板"按钮
```

### 5.4 历史记录数据流

```
翻译完成事件
        |
        v
[history 模块] 生成缩略图 + 保存记录到 SQLite
        |
        v
用户点击托盘"历史"
        |
        v
[前端 -> 后端] invoke("get_history_list")
        |
        v
[history 模块] 查询最近 50 条记录
        |
        v
[前端 HistoryView.vue] 列表展示
```

## 7. 安全架构

### 7.1 安全设计原则

- **最小权限**：Tauri IPC 命令采用白名单机制，前端仅能调用已注册的命令。
- **数据不落地**：截图图像仅存在于内存，不写入磁盘（历史缩略图除外）。
- **密钥隔离**：API 密钥不存储在配置文件中，通过 OS 凭据管理器独立存储。
- **通信加密**：所有 API 通信强制 HTTPS，验证 TLS 证书。

### 7.2 密钥存储架构

```
配置文件 (config.toml)
  |-- api_base_url
  |-- text_model
  |-- vision_model
  |-- target_language
  |-- default_mode
  |-- shortcuts
  |
  +-- api_key: "keyring://snaptranslate/api_key"  (引用，非明文)

OS 凭据管理器
  |-- service: "snaptranslate"
  |-- account: "api_key"
  |-- secret: <实际密钥值>
```

## 8. 部署架构

```

  |-- SnapTranslate.exe (主程序)
  |-- resources/
  |     |-- tessdata/          (OCR 语言数据)
```

## 9. 技术风险与缓解

| 风险编号 | 风险描述                                       | 影响 | 缓解措施                                         |
|---------|------------------------------------------------|------|--------------------------------------------------|
| R-01    | Wayland 下截图权限受限                          | 高   | 提前检测桌面协议，Wayland 下提示用户授权；优先支持 X11 |
| R-02    | Tesseract OCR 对某些字体/语言识别率低            | 中   | 提供多模态模式作为备选；允许用户切换模式            |
| R-03    | 大模型 API 响应格式不统一                        | 中   | 适配 OpenAI 兼容格式作为默认，提供自定义模板能力    |
| R-04    | 多贴图窗口资源占用过高                           | 中   | 限制最大贴图数量；未激活贴图降低渲染频率            |
| R-05    | 全局快捷键与其他软件冲突                         | 低   | 允许用户自定义快捷键；冲突时给出提示               |
